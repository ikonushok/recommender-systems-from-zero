# Popularity Baseline

## Что изучим

- зачем recommender-проекту нужен baseline;
- как превратить `MovieLens` из explicit ratings в простую top-K задачу;
- как сделать честный `leave-last-one-out` split без leakage;
- как работает `popularity baseline`;
- какие ограничения у такого подхода видны уже на первом запуске.

## Интуиция

`Popularity baseline` отвечает на очень простой вопрос:

`что рекомендовать, если вообще не персонализировать выдачу?`

Идея модели такая:

- посмотреть на train-данные;
- посчитать, какие объекты встречались чаще всего;
- рекомендовать пользователям один и тот же глобальный список, убирая из него уже seen items.

Это звучит слишком просто, но именно поэтому baseline так полезен.

Он нужен не для “красоты”, а для практики:

- даёт честную точку отсчёта;
- быстро вскрывает ошибки в данных и `split`;
- помогает не переоценивать более сложные модели.

Если сложная модель не может обыграть такой baseline, это почти всегда повод перепроверить постановку задачи, а не повод радоваться архитектуре.

## Мини-пример

Представим, что в train-данных самые частые фильмы такие:

1. `item_id = 318`
2. `item_id = 356`
3. `item_id = 296`
4. `item_id = 593`

Тогда без персонализации модель будет пытаться рекомендовать именно их.

Если пользователь уже видел `318` и `296`, выдача для него станет такой:

1. `356`
2. `593`
3. следующий фильм из общего popularity ranking

То есть baseline не понимает вкус пользователя. Он делает только одно полезное действие: не предлагает уже известные объекты и использует глобальную частоту как очень грубый сигнал релевантности.

## Как из explicit ratings получается top-K задача

В `MovieLens latest small` сигнал исходно относится к `explicit feedback`: пользователь явно ставит рейтинг от `0.5` до `5.0`.

Для первого baseline в проекте используется упрощение:

- положительным сигналом считаются оценки `>= 4.0`;
- более низкие оценки не превращаются автоматически в negative labels;
- задача формулируется как `какие фильмы пользователь с высокой вероятностью оценит положительно`.

В текущем notebook это даёт:

- `100,836` всех explicit interactions;
- `48,580` positive interactions;
- `positive_share = 0.4818`;
- `609` пользователей с positive feedback;
- `6,298` фильмов с positive feedback.

Это хорошая учебная постановка, потому что она позволяет аккуратно перейти от ratings к top-K рекомендациям без лишней сложности.

## Почему split здесь критичен

Даже для очень простой модели нельзя честно оценивать качество без правильного split.

В проекте используется упрощённый `leave-last-one-out`:

- для каждого пользователя последнее положительное взаимодействие уходит в `test`;
- остальные положительные взаимодействия остаются в `train`;
- у пользователя должен остаться хотя бы один объект в истории `train`.

Зачем это нужно:

- модель не видит будущее;
- проверка ближе к реальному сценарию “предсказать следующий интересный объект”;
- в `test` у каждого пользователя ровно один скрытый релевантный объект, поэтому метрики читаются особенно просто.

Именно на этом шаге особенно легко сделать leakage, если случайно смешать будущие и прошлые взаимодействия.

Важно понимать ещё одну границу этого core-сценария:

- здесь у нас есть только `train` и `test`;
- отдельного `validation`-набора пока нет.

Это сознательное учебное упрощение.

Что это означает на практике:

- для baseline и первых моделей такой split удобен и прозрачен;
- но любой последующий подбор гиперпараметров на тех же данных методически уже начинает подглядывать в `test`.

Поэтому в `advanced`-части маршрут станет строже.

Там для задач feature engineering и tuning мы перейдём к схеме:

- `train`: всё, кроме двух последних positive interactions пользователя;
- `val`: предпоследнее positive interaction;
- `test`: последнее positive interaction.

## Что делает popularity baseline

После split модель обучается очень просто:

1. считает число positive interactions для каждого `item_id` в `train`;
2. сортирует фильмы по убыванию популярности;
3. для каждого пользователя возвращает top-K из этого глобального списка;
4. уже seen items исключаются.

Никаких user-features и item-features здесь нет.

Это важно понимать правильно:

- baseline может быть полезным;
- baseline не является персонализированной моделью;
- baseline не должен притворяться collaborative filtering или content-based подходом.

## Что получилось

На текущем запуске baseline дал такие результаты:

- `HitRate@10 = 3.95%`;
- `unique_recommended_items = 61`;
- `catalog_coverage = 0.97%`.

Как это интерпретировать:

- модель действительно иногда угадывает скрытый test-фильм;
- но выдача очень узкая и head-heavy;
- большинство рекомендаций строится вокруг ограниченного числа самых популярных объектов.

Это и есть ожидаемое поведение popularity baseline:

- он может быть surprisingly strong как отправная точка;
- но быстро упирается в отсутствие персонализации.

## Ограничения метода

У popularity baseline есть очевидные ограничения:

- он почти не различает пользователей между собой;
- усиливает `popularity bias`;
- плохо покрывает каталог;
- почти не помогает новым и нишевым объектам;
- не решает cold-start содержательно, а только прячет проблему под head-объектами.

Но именно эти ограничения и делают baseline полезным.

Если более сложная модель:

- не даёт выигрыша по метрикам;
- не улучшает coverage;
- или выглядит “умной” только на словах,

то baseline помогает быстро это заметить.
