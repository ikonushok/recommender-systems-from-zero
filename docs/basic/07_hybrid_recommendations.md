# Hybrid Recommendations

## Что изучим

- зачем recommender systems комбинируют несколько источников сигнала;
- почему одна модель редко бывает лучшей во всех аспектах сразу;
- как устроен простой hybrid на `weighted interleaving`;
- как сравнивать hybrid с его компонентами;
- почему hybrid нужно настраивать осторожно, а не “магически смешивать всё со всем”.

## Интуиция

После нескольких базовых моделей обычно оказывается, что у каждой есть своя сильная сторона.

В нашем маршруте это видно очень хорошо:

- `popularity baseline` стабилен, но почти не персонализирует;
- `content-based` сильно расширяет coverage, но на одних жанрах даёт слабое качество;
- `item-based collaborative filtering` лучше по метрикам качества, но зависит от плотности взаимодействий и плохо справляется с холодным стартом.

Отсюда и появляется идея hybrid-подхода:

`не выбирать одну модель навсегда, а аккуратно комбинировать разные сигналы`

Но здесь есть важная оговорка:

- hybrid не обязан автоматически стать лучшей моделью;
- плохо настроенный hybrid может оказаться хуже сильной компоненты;
- смысл гибрида в аккуратном компромиссе, а не в механическом смешивании.

## Мини-пример

Пусть для одного пользователя модели дают такие top-списки:

`item-based collaborative filtering`:

1. `2918`
2. `858`
3. `296`
4. `1036`

`content-based`:

1. `5657`
2. `6990`
3. `26340`
4. `108932`

Тогда hybrid не должен:

- просто полностью копировать collaborative-модель;
- но и не должен ставить слабый content-based сигнал в верх списка слишком агрессивно.

В текущем notebook итоговый hybrid для этого пользователя выглядит так:

1. `2918`
2. `858`
3. `296`
4. `1036`
5. `589`
6. `2762`
7. `1200`
8. `1221`
9. `5657`
10. `6990`

Это уже осмысленный компромисс:

- верх списка сохраняет сильный collaborative-сигнал;
- content-based часть не исчезает полностью;
- hybrid остаётся отдельной моделью, а не копией одной компоненты.

## Почему hybrid вообще нужен

Hybrid-модель особенно полезна, когда разные компоненты решают разные подзадачи:

- одна модель лучше по метрикам качества;
- другая лучше покрывает каталог;
- одна сильнее на head-объектах;
- другая лучше даёт разнообразие или объяснимость.

Тогда задача гибрида не обязательно “обогнать всех по одной метрике”, а:

- не разрушить сильную компоненту;
- добавить второй источник сигнала там, где он полезен;
- получить более устойчивое поведение на разных типах пользователей и объектов.

## Какой hybrid используется в проекте

В текущем core-маршруте используется простой `weighted interleaving`.

Идея такая:

1. каждая модель строит свой ранжированный список;
2. hybrid чередует элементы этих списков;
3. дубликаты удаляются;
4. более сильная модель получает больший вес и идёт раньше в каждом раунде.

Для нашего учебного сценария это хороший выбор, потому что:

- логика легко объясняется;
- можно глазами посмотреть, как смешиваются списки;
- не нужна сложная обучаемая fusion-схема.

## Что пришлось сделать, чтобы hybrid стал разумным

Это важный учебный момент: хороший hybrid получился не с первого раза.

Наивный вариант был слишком агрессивен к слабому `content-based` сигналу и портил верх списка.

Чтобы исправить это, было сделано две вещи:

1. Сначала не просто меняли веса, а поменяли приоритет порядка.
   Более сильная `item-based collaborative filtering` модель теперь идёт раньше в каждом раунде interleaving.

2. После этого уменьшили долю слабого `content-based` сигнала.
   В текущем notebook веса настроены так, чтобы collaborative-модель занимала большую часть `top_k`, а content-based часть добавлялась осторожно.

Это очень полезный вывод для новичка:

- гибрид нельзя оценивать только по красивой идее;
- важно смотреть, что реально происходит в верхней части списка;
- даже простая схема fusion требует настройки.

## Что получилось

На текущем запуске результаты такие:

- `popularity baseline`: `precision@10 = 0.0039`, `recall@10 = 0.0395`, `map@10 = 0.0162`, `ndcg@10 = 0.0218`;
- `content-based`: `precision@10 = 0.0005`, `recall@10 = 0.0049`, `map@10 = 0.0012`, `ndcg@10 = 0.0020`;
- `item-based collaborative filtering`: `precision@10 = 0.0064`, `recall@10 = 0.0641`, `map@10 = 0.0191`, `ndcg@10 = 0.0295`;
- `hybrid interleaving`: `precision@10 = 0.0054`, `recall@10 = 0.0543`, `map@10 = 0.0180`, `ndcg@10 = 0.0265`.

Это хороший учебный результат, потому что:

- hybrid не стал лучшей моделью по всем метрикам;
- но он заметно лучше `popularity baseline`;
- и остаётся довольно близок к сильной collaborative-компоненте, особенно по `MAP@10` и `NDCG@10`.

То есть hybrid в этом проекте выполняет свою задачу:

- не разрушает сильный collaborative-сигнал;
- добавляет второй источник рекомендаций;
- остаётся отдельной моделью, а не полным дублем компоненты.

## Как это интерпретировать правильно

Из этой главы полезно вынести несколько вещей:

- hybrid не обязан быть “чемпионом по всем цифрам”;
- гибрид полезен, если он сохраняет большую часть качества сильной модели и даёт дополнительный сигнал;
- плохая компонента может испортить hybrid, если дать ей слишком много влияния;
- настройка порядка и весов важна не меньше самой идеи fusion.

Именно поэтому в реальных recsys-системах hybrid-подходы встречаются часто:

- не потому что “смешать модели” всегда автоматически хорошо;
- а потому что аккуратно настроенная комбинация сигналов часто устойчивее одной модели.

## Ограничения метода

Даже такой простой hybrid имеет ограничения:

- веса fusion выбраны вручную, а не обучены;
- если одна из компонент очень слабая, она может только шуметь;
- hybrid не отменяет cold-start ограничения исходных моделей;
- при большом числе моделей схему fusion нужно калибровать аккуратнее;
- offline улучшение гибрида не гарантирует product uplift без онлайн-проверки.

То есть hybrid — это не “волшебная надстройка”, а ещё один слой инженерных и методических решений.
