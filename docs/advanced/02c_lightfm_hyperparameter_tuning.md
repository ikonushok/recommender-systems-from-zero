# LightFM Hyperparameter Tuning

## Что изучим

- почему после улучшения признаков следующий шаг — тюнинг модели, а не немедленный переход в `Neural CF`;
- какие гиперпараметры `LightFM` сильнее всего влияют на ranking-качество;
- как подбирать конфигурацию на учебном датасете без превращения главы в большой MLOps-пайплайн;
- как сравнивать лучшую tuned-версию с `ALS`.

## Зачем эта глава после 02b

После feature engineering у нас уже есть более сильная hybrid-модель.

Теперь честный следующий вопрос такой:

`достаточно ли признаков, или модель ещё недоиспользует их из-за неудачных гиперпараметров?`

Отдельная tuning-глава нужна, чтобы не смешивать две разные оси улучшения:

- качество признаков;
- качество настройки модели.

## Постановка главы

Сохраняем:

- тот же датасет;
- тот же `train / validation / test` split;
- тот же engineered feature set из `02b`, который оказался сильнейшим по общему профилю метрик;
- те же метрики и candidate universe.

Меняем только:

- конфигурацию `LightFM`.

Роли наборов здесь уже нужно разделять явно:

- `train` используется для обучения;
- `validation` используется для выбора лучшей конфигурации в `Optuna`;
- `test` остаётся held-out набором для финальной проверки после tuning.

## Что должно быть в notebook

- фиксированная baseline-конфигурация из `02b`;
- поиск конфигураций через `Optuna`, а не ручной перебор;
- одинаковая процедура оценки для всех trial'ов на `validation`;
- таблица результатов `Optuna`;
- встроенные графики истории оптимизации и важности гиперпараметров;
- выбор лучшей конфигурации по явно заданной метрике;
- финальное сравнение `ALS` vs `LightFM tuned` на held-out `test`.

## На что смотреть в первую очередь

Для учебного `LightFM` обычно достаточно начать с:

- `loss`: `warp`, `bpr`, `warp-kos`;
- `no_components`: `16`, `32`, `64`;
- `epochs`: умеренный рост без взрыва времени;
- `item_alpha` и `user_alpha`;
- при необходимости `learning_rate`.

Главная цель главы:

- получить модель, сопоставимую с `ALS` или лучше него хотя бы по ключевым ranking-метрикам;
- при этом честно зафиксировать, если выигрыш идёт не по всем метрикам сразу.

## Основные риски

- сделать слишком большой grid и утонуть во времени прогона;
- подгонять конфигурацию под один lucky-run без объяснения;
- выбрать лучшую модель без явного критерия;
- перепутать gain от tuning и gain от feature engineering.

## Что считается готовностью главы

- notebook показывает, что `LightFM` можно усиливать и через признаки, и через настройку;
- лучшая конфигурация выбирается по `validation`, а затем сравнивается с `ALS` на held-out `test`;
- читателю ясно, какие параметры дали прирост и почему;
- после этой главы переход к `Neural CF` выглядит осмысленным, а не преждевременным.
