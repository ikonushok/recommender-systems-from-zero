# Two-Tower Models

## Что изучим

- чем retrieval-задача отличается от pair scoring;
- зачем в recsys разделять candidate generation и последующее ranking;
- как работает интуиция `two-tower` модели;
- почему для этой главы уже нужен более богатый по признакам датасет;
- как читать retrieval-результат, не путая его с финальным ranking-качеством.

## Зачем эта глава после Neural CF

`ALS`, `LightFM` и `NeuMF` помогали думать в одной и той же логике:

- есть пользователь;
- есть объект;
- нужно оценить, насколько хороша пара `user-item`.

Но в реальной системе часто есть ещё один вопрос:

`как вообще быстро выбрать разумных кандидатов из большого каталога?`

На этот вопрос и отвечает глава про `two-tower` и retrieval.

Это очень важный переход маршрута:

- до этого мы в основном думали про scoring;
- теперь начинаем думать про stages внутри рекомендательной системы.

Именно поэтому `03` — это не просто “ещё одна модель”, а смена самой постановки.

## Главная идея retrieval

В pair-scoring сценарии модель обычно отвечает на вопрос:

`насколько подходит этот item данному пользователю?`

В retrieval-сценарии вопрос другой:

`как быстро найти небольшой shortlist разумных кандидатов среди очень большого каталога?`

Это меняет всё:

- user и item начинают кодироваться отдельно;
- финальный score чаще нужен для nearest-neighbor поиска в embedding space;
- цель модели — не сразу построить финальный порядок для витрины, а сначала сузить candidate universe.

Именно поэтому retrieval и ranking нельзя смешивать в одну неразличимую сущность.

## Почему именно two-tower

`Two-tower` — хороший учебный способ показать retrieval-мышление, потому что модель естественно разделяет:

- пользовательское представление;
- представление объекта.

Дальше retrieval работает так:

- user tower строит embedding пользователя;
- item tower строит embedding объекта;
- кандидаты ищутся по близости этих embedding'ов.

Это очень наглядная архитектура для момента курса, где нужно показать:

- почему candidate generation — это отдельный слой;
- и почему его нельзя просто считать “тем же ranking, только по-другому”.

## Почему здесь уже можно сменить датасет

До этого `MovieLens` был удобен:

- он хорошо подходит для `ALS`, `LightFM`, `NeuMF`;
- там основной сигнал — это `user-item interactions`;
- на нём легко держать одну и ту же offline-постановку.

Для `two-tower` задача другая.

Чтобы retrieval был наглядным, нужны богатые item-features.

В `MovieLens latest small` таких признаков мало:

- `title`;
- `genres`;
- немного `tags`.

На `Amazon Reviews 2023 / All_Beauty` сценарий естественнее:

- есть `title`;
- есть `description`;
- есть `features`;
- есть `categories`;
- есть `store`.

Поэтому user/item text-представления там имеют учебный смысл.

Это одна из немногих глав advanced-маршрута, где смена датасета не мешает, а наоборот помогает объяснить новую идею.

## Постановка главы

Эта глава специально меняет уже не только модель, а сам тип задачи.

Здесь мы:

- строим text-rich item representation;
- собираем user profile из его истории;
- учим отдельные user/item towers;
- сравниваем retrieval-модель хотя бы с `popularity baseline`;
- отдельно проговариваем границу: retrieval ещё не равен final ranking.

При этом chapter остаётся учебной:

- без ANN-индексов;
- без distributed training;
- без production retrieval stack.

Это нужно сказать прямо, чтобы не возникло ложного ощущения, будто одна глава про `two-tower` уже делает проект production-ready.

## Что должно быть в notebook

- компактный воспроизводимый data slice;
- явные user/item признаки;
- `train / validation / test` по временной логике на пользователя;
- text-based item features и user history profiles;
- обучение учебной `two-tower` retrieval модели на `PyTorch`;
- сравнение хотя бы с `popularity baseline`;
- генерация кандидатов для пользователя;
- sanity-check на retrieval-качестве;
- отдельная интерпретация, чем retrieval отличается от final ranking.

## Почему здесь важен временной split

Даже в retrieval-главе split остаётся критическим.

В notebook используется та же честная логика:

- последнее positive interaction пользователя уходит в `test`;
- предпоследнее — в `validation`;
- ранняя история остаётся в `train`.

Это важно по двум причинам:

- retrieval нельзя оценивать на данных из будущего;
- user profile должен строиться только по доступной на тот момент истории.

Отдельно полезно смотреть на:

- `validation_new_item_share_vs_train`;
- `test_new_item_share_vs_train`.

Это честно показывает долю объектов, которые retrieval-модель вообще не могла знать по train-каталогу.

## Почему `new_item_share_vs_train` здесь особенно важен

Это одна из ключевых особенностей главы.

Если отложенный объект вообще не встречался в train-каталоге, retrieval-модель:

- не умеет его вытаскивать через train-item embeddings;
- и её нельзя честно обвинять в том, что она “не нашла” этот item внутри известного каталога.

В notebook это проговаривается прямо:

- часть validation/test объектов может быть полностью новой для train;
- это честное ограничение retrieval-постановки.

Для этой главы это важнее, чем для `ALS`/`LightFM`, потому что retrieval здесь особенно явно живёт внутри ограниченного train-catalog universe.

## Как правильно читать результат two-tower

Здесь очень легко сделать две ошибки.

Ошибка 1:

`если two-tower обыгрывает popularity, значит модель уже готова к продакшену`

Ошибка 2:

`если absolute ranking-метрики низкие, значит retrieval не сработал`

Корректнее так:

- `popularity` — это очень слабый retrieval baseline;
- прирост над ним показывает, что user/item representations действительно дают полезный сигнал;
- почти полный `catalog_coverage` важен сам по себе: retrieval перестаёт крутиться вокруг одних и тех же head-объектов;
- absolute `MAP@10` и `NDCG@10` могут оставаться низкими, потому что задача шумная, train-срез небольшой, а часть target items вообще недоступна для retrieval из train-каталога.

То есть результат главы нужно читать прежде всего как:

- доказательство полезности retrieval-stage;
- а не как доказательство “финальное качество уже хорошее”.

## Retrieval не равен final ranking

Это, вероятно, главный вывод всей главы.

Даже если `two-tower` даёт хороший shortlist, это ещё не означает:

- что порядок внутри него уже идеален;
- что на витрину можно сразу отдать именно этот список;
- что можно отказаться от следующего ranking-слоя.

В реальной системе `two-tower` часто живёт так:

- candidate generation через embeddings;
- потом ranking на richer features;
- потом re-ranking или бизнес-правила.

Именно поэтому после retrieval-главы later в маршруте появляется отдельная тема `ranking / LTR`.

## Что здесь сознательно не делаем

В рамках проекта мы не будем:

- строить ANN-индексы;
- поднимать distributed retrieval serving;
- обучать большие dual-encoder системы на полном Amazon-scale;
- делать production optimization по latency и memory.

Это было бы архитектурно тяжело и учебно вредно для текущей цели главы.

Наша задача скромнее:

- понять retrieval-мышление;
- увидеть, как user/item representations работают отдельно;
- честно отделить retrieval от final ranking.

## Основные риски

- сделать вид, что `two-tower` заменяет весь recommender pipeline;
- перегрузить главу инфраструктурой вместо идеи retrieval;
- взять слишком большой срез `Amazon` и потерять воспроизводимость;
- скрыть ограничение по новым `item_id`, которых не было в train;
- не объяснить, чем retrieval-метрики и top-K ranking-метрики соотносятся друг с другом;
- не проговорить, что retrieval работает только внутри известного train-каталога.

## Что считается готовностью главы

- новичку понятно, зачем нужна отдельная retrieval-стадия;
- выбор `Amazon Reviews 2023 / All_Beauty` обоснован;
- различие между pair scoring и retrieval сформулировано простым языком;
- ограничения учебной реализации прямо названы: нет ANN, нет distributed serving, нет решения для completely new items;
- роль `new_item_share_vs_train` объяснена явно;
- глава естественно подводит к теме последовательностей и более контекстных сценариев.
