# LightFM Feature Engineering

## Что изучим

- почему слабый результат hybrid-модели не всегда означает слабость самого класса модели;
- как усиливать `LightFM`, не меняя датасет, split и candidate universe;
- какие признаки в `MovieLens` можно добавить поверх `genres` без лишней сложности;
- как читать прирост качества именно как эффект feature engineering.

## Зачем эта глава после 02a

В `02a` мы уже увидели сам принцип hybrid factorization:

- есть взаимодействия пользователей с объектами;
- к ним добавляются признаки объектов;
- `LightFM` начинает использовать оба источника сигнала.

Но одного этого шага недостаточно.

Если модель на голых `genres` уступает `ALS`, следующий честный вопрос такой:

`слабая сама модель или слабые признаки?`

Именно на этот вопрос отвечает отдельная глава про feature engineering.

## Постановка главы

Здесь важно не подменить одну проблему другой.

Поэтому сохраняем:

- тот же `MovieLens latest small`;
- тот же `implicit` сигнал;
- тот же `train / validation / test` split;
- ту же логику top-K оценки.

Меняется только одно:

- набор item-features для `LightFM`.

## Что должно быть в notebook

- напоминание, что baseline `LightFM` на `genres` уже есть из `02a`;
- добавление richer-feature набора поверх `genres`;
- минимум один честный train-time-safe источник новых признаков, например `tags`;
- простые `title`-признаки из названия фильма;
- повторное обучение `LightFM` на том же split;
- диагностика coverage / sparsity / редких токенов до обучения;
- pruning части шумных признаков и повторное сравнение метрик;
- сравнение `lightfm_genres` vs `lightfm_engineered` и при необходимости против `ALS`;
- интерпретация: где прирост идёт от признаков, а где уже упирается в саму модель.

## Рекомендуемые признаки

Минимальный реалистичный набор для `MovieLens`:

- `genres`;
- простая `decade`-фича из названия;
- токены из `title`;
- `tags.csv`, но только до train cutoff, чтобы не было leakage из будущего.

Важно:

- не превращать главу в большой tutorial по preprocessing;
- не использовать признаки, которые недоступны к моменту рекомендации;
- не делать выводы сильнее, чем позволяет маленький `MovieLens`.

## Основные риски

- протащить future information через `tags`;
- расползтись в слишком длинный preprocessing-блок;
- начать удалять признаки без проверки их частоты и покрытия;
- спутать эффект feature engineering с эффектом случайного тюнинга;
- забыть сравнить engineered-версию не только с `ALS`, но и с предыдущим `LightFM`.

## Что считается готовностью главы

- видно, что `LightFM` можно усиливать через признаки, а не только через смену модели;
- соблюдён тот же split и candidate universe;
- признаки построены без leakage;
- notebook даёт честный ответ, улучшил ли feature engineering hybrid-модель.
